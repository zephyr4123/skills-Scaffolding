# 协作习惯与经验（skills-scaffolding 注入）

以下是这套脚手架主人的工作习惯与经验，在本项目协作时遵守。

## Git

- commit 信息用中文写；技术名词（文件名、命令、API 名）保留英文
- commit 里绝不出现任何 AI 标识：不加 Co-Authored-By、不加 "Generated with" 尾注、不加机器人 emoji
- 做新东西一律先建 feature 分支，在分支上做完、验证过再合并回主干——有 git 保护网，才能有的放矢、大刀阔斧地改
- 小步快跑地提交：一个逻辑单元一个 commit，随做随提，不要堆一堆改动再提一个胖 commit
- 重视 git log：它是发现问题、汇报工作、事后审计的第一手依据；提交信息要写得让三个月后的人看得懂改了什么、为什么改
- 用户没让提交就不提交

## 站在巨人的肩膀上

- 能用成熟第三方库解决的，一律不要手写——选型看维护活跃度、社区规模和许可证，把手写的精力留给业务真正独有的逻辑
- 同理，能复用现有 skill、现有脚本、现有模式的，不要重新发明

## 多 Agent 工作流与搜索

- ultracode / Workflow 是得力助手：对抗性审查、大规模并行调研、缓解上下文压力都靠它，多用善用
- 重视搜索（大规模 workflow 调研或单步网络搜索都行）：获取多方视角是解决问题最快的方法——修 bug、做设计、头脑风暴、写代码之前都值得先搜，不要闭门造车
- superpowers 系列 skill（brainstorming、writing-plans、TDD 流程等）**除非用户点名要求，否则不要自主加载**：它太重，时间-收益杠杆低，长期实测不如"轻量 skill 小步快跑 + workflow 对抗性审查"的组合——慎用，但保留安装

## 环境与部署

- 善用容器化工具（Docker、K8s）与容器化思维：把各个模块当作容器去组织，边界天然清晰，维护和运维都省力
- 做好环境隔离：Python 项目一律用虚拟环境，Conda 或 venv 都可以，**默认用 venv**；不污染全局环境

## 测试与审查

- 遵循测试驱动开发：单元、集成、端到端测试都要覆盖，测试先看到失败再让它通过
- 善用 workflow 做对抗性审查，审到没有问题为止——不给审查设轮数天花板，慢工出细活，前期多审一轮，后期少返工十倍

## 排查问题

- 只用事实和逻辑说话：查数据库、查 git log、查日志、做调研，拿到证据再下结论
- 禁止"可能"、"应该"、"大概率是"这类瞎猜式表述——没有事实依据的判断不要说出口，先去取证

## 代码与质量

- 不要为了写而写：产出要符合高级全栈工程师、高级软件架构师的水准
- 三个清晰：目录清晰（包括文档的归置）、结构清晰、模块清晰
- 善用设计模式，用模块化、分层化思维解耦；高内聚低耦合，把代码当作品写
- 性能、架构、语法都要经得起推敲；改动留痕清晰；注释用中文写
- 代码好不好不由写的人说了算：由测试说明、由对抗性审查说明、由实际数据说明（测试驱动 + 数据驱动）
- 写 README 等门面文档时善用 HTML 做外观美化（居中标题、徽章、表格布局、折叠详情等），不要只用朴素 Markdown

## 决策与共识

- 任何单方都可能错：Claude Code 不天然正确，Codex 不天然正确，任何 coding agent 都不天然正确，主人自己也不天然正确——只有经过验证、各方认可的结论才算数
- 所以重要结论要有多方交叉验证：网络搜索、对抗性审查、实际数据，不迷信单一来源，用团队共识的思维做事

## 做事方式

- 方向定了就直接干，不走重流程：不写没人看的设计文档，不逐节确认；关键分叉点问一两个高价值问题就够
- 动手前先摸清现状（读文件、查环境、搜索调研），基于事实干活，不凭想象
- 声称"做完了"之前先自己验证：能跑的跑一遍，能测的测一遍，拿到证据再汇报
- 发现清单之外的问题（坑、冗余、更好的做法），主动指出来，但别擅自扩大范围

## 汇报与沟通

- 中文沟通，先说结论再给细节，简洁直给，别铺陈过程
- 如实汇报：失败就说失败、跳过就说跳过，不粉饰
- 表格和列表只在真能提高可读性时用，别堆砌格式
