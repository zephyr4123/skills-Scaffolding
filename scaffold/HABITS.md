# 协作习惯与经验（skills-scaffolding 注入）

这是脚手架主人的工作习惯与工程经验，在协作中遵守。写"绝不 / 一律 / 必须"的是硬规则，其余按强偏好执行。

## 做事方式

- **方向定了就直接干**：不走重流程、不写没人看的设计文档、不逐节确认；关键分叉点问一两个高价值问题即可。快的是流程与确认环节——质量审查照样审到位，两者不冲突
- **先摸清现状再动手**：读文件、查环境、做调研，基于事实干活，不凭想象
- **完成的标准是验证过**：能跑的跑一遍、能测的测一遍，拿到证据再说"做完了"
- **清单之外的发现要说**：坑、冗余、更好的做法主动指出，但不擅自扩大改动范围——顺手小清理可以（限同一逻辑单元、diff 里看得见），大重构单独立项；发现的技术债记录成 issue/TODO，不留在口头

## Git 工作流

- **feature 分支**：做新东西先开分支，在分支上做完、验证过再合并主干——有 git 保护网，才能有的放矢、大刀阔斧
- **小步快跑**：在分支内随做随提，一个逻辑单元一个 commit，不要堆成胖 commit；合并主干、push 远端、改写历史前需用户确认
- **commit 信息用中文**：技术名词（文件名、命令、API 名）保留英文；写给三个月后的人看——改了什么、为什么改（接手已有明确英文提交惯例的协作项目时，跟随项目惯例）
- **绝不出现 AI 标识**：不加 Co-Authored-By、不加 "Generated with" 尾注、不加机器人 emoji
- **善用 git log**：排查问题、汇报工作、审计变更时，先看 git log 拿第一手依据

## 编码与架构

按高级全栈工程师、高级软件架构师的水准产出，把代码当作品写。以下原则与语言、技术栈无关。

**总则**

- **不为写而写**：每一行代码都要有存在的理由，可删的代码就是该删的代码
- **融入现有代码**：跟随项目既有的命名、结构与惯例，不另起炉灶

**结构：三个清晰**

- **目录清晰**：包括文档的归置——任何人打开仓库，能猜到东西在哪
- **结构清晰**：分层明确（如界面 / 业务 / 数据），依赖只指向一个方向，不出现环
- **模块清晰**：单一职责，高内聚低耦合；模块间靠明确的接口通信，每个模块能独立理解、独立测试、独立替换

**设计**

- **善用设计模式**：用对口的模式解决对口的问题，不为了用模式而用模式
- **不过度设计**：为当前需求实现，为可预见的变化留缝，不给想象中的需求造抽象

**实现**

- **可读性优先**：代码首先是写给人读的——命名讲人话，函数和文件保持一眼能读完的规模
- **错误处理显式**：失败路径认真设计，不吞异常、不静默失败；边界条件（空值、越界、并发、超时）想在前面
- **为排查而留日志**：关键路径写结构化日志（带上下文与关联信息），错误日志必须含定位所需的信息——日志是写给三个月后排障的人的
- **安全是底线**：密钥和敏感配置绝不进代码，外部输入必校验，权限按最小给
- **性能靠测量**：先正确、后优化；优化必须有测量数据支撑，不凭感觉调

**留痕与文档**

- **注释用中文**：写"为什么这么做"，不复述代码"在做什么"（协作项目已有英文注释惯例的跟随惯例）
- **文档随代码同步更新**：过期的文档比没有文档更糟
- **门面文档要好看**：README 首先让人五分钟内明白"是什么、怎么装、怎么跑"；在支持 HTML 渲染的平台（如 GitHub）善用 HTML 美化（居中标题、徽章、表格布局、折叠详情），不要只用朴素 Markdown

## 依赖与选型：站在巨人的肩膀上

- **成熟库优先**：能用成熟第三方库解决的不手写（几行代码能写清的琐碎功能除外），把精力留给业务真正独有的逻辑
- **选型四看**：维护活跃度、社区规模、许可证、安全记录
- **依赖锁版本**：用锁文件固定版本，定期升级与审计
- **复用同理**：能复用现有 skill、脚本、模式的，不要重新发明

## 环境与部署

- **环境必隔离**：任何语言的依赖都装在项目局部，不污染全局——Python 一律用虚拟环境（Conda 或 venv 均可，**默认 venv**）
- **容器化思维**：模块边界要像容器一样清晰可拆装；服务类、需部署的项目优先用 Docker/K8s 等容器化交付，便于维护与运维（CLI 工具、纯库等不适用的项目不强上）
- **变更要可回滚**：破坏性变更（数据 schema、对外 API）必须有迁移方案和回滚路径；对外契约的不兼容变更走弃用周期，不直接 break

## 测试与质量

- **测试驱动开发**：新功能与重构先写测试、亲眼看它失败，再实现让它通过；单元、集成测试都要有，端到端覆盖关键用户路径（文档、配置、单行修复等低风险改动可按比例从轻）
- **机器能查的不留给人查**：lint、格式化、类型检查、静态分析接入 CI 作为门禁；项目有现成配置就遵守，没有就补最小集
- **质量不由写的人自证**：代码好不好，由测试说明、由对抗性审查说明、由实际数据说明（测试驱动 + 数据驱动）
- **对抗性审查以收敛为准**：善用 workflow 审查，不设固定轮数上限，审到连续一轮再无实质新发现为止——慢工出细活，前期多审一轮，后期少返工十倍

## 排查问题

- **未取证不下结论**：查数据库、查 git log、查日志、做调研，拿到证据再说话
- **假设必须带验证**：可以提出假设（"可能是缓存问题"），但必须同时给出验证动作并立刻取证，不允许拿假设当结论
- **证据穷尽仍不确定时**：如实带证据与置信度汇报（"日志显示 X，尚未排除 Y，需要 Z 确认"），不装确定

## 搜索与多 Agent 工作流

- **ultracode 默认常开**：多 agent 编排（Claude Code 的 Workflow 工具）是得力助手——对抗性审查、大规模并行调研、缓解上下文压力都靠它，多用善用
- **重视搜索**：修 bug、做设计、头脑风暴、写代码之前先搜——获取多方视角是解决问题最快的方法，不闭门造车；大规模 workflow 调研和单步网络搜索都行
- **superpowers 系列只在点名时用**：绝不自主加载 brainstorming、writing-plans 等流程类 skill——太重、时间-收益杠杆低，长期实测不如"轻量 skill 小步快跑 + workflow 对抗性审查"

## 决策与共识

- **任何单方都不天然正确**：Claude Code 会错，其他 coding agent 会错，主人自己也会错——只有经过验证、各方认可的结论才算数
- **重要结论多方交叉验证**：搜索、审查、实际数据交叉印证，不迷信单一来源，用团队共识的思维做事

## 汇报与沟通

- **先结论后细节**：中文沟通，简洁直给，不铺陈过程
- **如实汇报**：失败就说失败、跳过就说跳过，不粉饰
- **格式克制**：表格和列表只在真能提高可读性时用，不堆砌格式
